## High-level assessment
- **Positive:** The frame-capture feature is well covered by tests and addresses prior race/loss issues with lock-guarded callback state and incremental polling semantics.
- **Risk:** There are still a few correctness/safety/operability gaps around input validation and long-run memory behavior.

---

## Findings

## 1) Medium: `/api/train/frames` accepts negative `since`, enabling expensive full-buffer replays repeatedly

**Where**
- `src/rl_framework/gui/app.py` parses `since` with `int(...)` but does not clamp lower bound.
- `src/rl_framework/training/frame_capture_callback.py` filters with `frame_index >= since`.

**Why it matters**
- A negative `since` (e.g. `-999999`) always returns the full ring buffer. Repeated polling with negative values can force avoidable payload and encoding overhead and degrade UI/API responsiveness.

**Recommendation**
- Clamp `since = max(0, parsed_value)` in the route handler.

---

## 2) Medium: Client-side frame buffer is unbounded and can grow indefinitely during long runs

**Where**
- `src/rl_framework/gui/static/app.js` appends every new frame into `frameData` and never evicts old entries.

**Why it matters**
- For long training sessions, browser memory usage grows without bound (each frame includes base64 JPEG data), eventually causing UI lag or tab crashes.

**Recommendation**
- Bound client buffer size (e.g., keep last N frames such as 500–2000).
- If full history is needed, persist server-side and page-window on demand instead of retaining all base64 in browser memory.

---

## 3) Low: Broad exception swallowing in frame capture can hide persistent rendering failures

**Where**
- `src/rl_framework/training/frame_capture_callback.py` wraps `_capture_frame` in broad `except Exception` and only logs when `verbose >= 1`.

**Why it matters**
- In default/quiet runs, persistent frame pipeline failures are silent, reducing debuggability and making visualization appear flaky.

**Recommendation**
- Keep broad guard for training safety, but increment a failure counter and expose it through training status/metrics for observability.

---

## What looks good
- Thread safety in callback frame deque access with explicit lock.
- Incremental `since` API semantics to avoid always re-sending all frames.
- Tests for frame filtering, episode counting, shape handling (including 2D/grayscale and RGBA), and API forwarding behavior.

## Suggested follow-up work (small, reviewable)
1. Clamp `since` to non-negative in API.
2. Cap `frameData` on frontend.
3. Add minimal observability field for capture failures in status endpoint.
